# Playbook - Harness Engineering

> 多 Agent 协作开发流程。适用于正式交付、跨模块改动。
> 来源：OpenAI Harness Engineering + 实战总结

## 状态：实践中 🟢

## 适用场景

- 正式交付、需要质量保障
- 跨模块、跨工程改动

## 核心理念

- **Humans steer, Agents execute** — 人掌舵，Agent 干活
- **Agent 搞砸时，补护栏而非重试**
- **职责分离本身就是质量保障**
- **发现问题的人不能自己修**

## 七角色分工

| 角色 | 职责 |
|------|------|
| PM Orchestrator | 流程总控、阶段协调、前进/回退决策 |
| Requirement Analyst | 需求拆解、多义性分析、验收标准 |
| Solution Architect | 模块划分、接口定义、风险预判 |
| Gate Reviewer | 开发前多维度审查，有阻塞项不许开工 |
| Developer Agent | 按方案落地，唯一写代码的角色 |
| Code Reviewer | 逻辑漏洞、需求遗漏、设计偏离 |
| QA Tester | 测试用例、缺陷分类 |

## 流程

```
需求分析 → 方案设计 → 闸门评估 → 开发 → 代码评审 → 测试 → 交付
                          ↓                  ↓          ↓
                     回退到需求/方案      回退到开发   回退到开发
```

## 三层验证

| 层 | 作用 |
|----|------|
| 流程 Rule | 保证 AI 不跳过验证步骤 |
| Skill | 封装验证操作 |
| Script | 机械化检测，不靠 AI 记忆 |

**原则：能用机器查的就别靠 AI 记。**

## 流程重量匹配

| 任务规模 | 建议 |
|----------|------|
| 大需求/跨工程 | 完整七阶段 |
| 中等/单工程 | 可省闸门 |
| 小 Bug | 根因→开发→评审→测试 |
| 十行以内 | 直接改 + 验证 |
